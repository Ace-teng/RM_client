# RoboMaster 自定义客户端 — UI 设计需求说明（供设计稿对照）

**用途说明：** 当前客户端仅将功能以列表/表单形式堆叠在界面上，未做信息层级、视觉主次、布局与交互设计，属于不合格产品。本文档列出**界面上必须存在的元素、区域划分、按角色的差异、以及交互与状态**，不规定具体视觉风格与像素级布局，供 UI/UX 设计师绘制正式设计稿。设计稿完成后，开发将**严格按设计稿**实现界面与布局。

---

## 一、产品与使用场景（设计前提）

- **谁用**：比赛时操作间内的操作手（英雄、工程、步兵1、步兵2、云台手等不同角色）。
- **在哪用**：操作间内一台 PC 或带网口的平板，有线接赛事网络；可能单屏也可能多屏。
- **主要任务**：看机器人图传画面、看己方/对方状态、快速发赛事指令、看战术建议与诊断信息；不同角色关注的信息侧重不同。
- **约束**：先保证与官方选手端功能对等（T0），再客制化；信息来自同一数据源，仅展示方式按角色切换。

---

## 二、整体布局与视觉层级（需在设计稿中体现）

- **主视觉**：图传画面应为主视觉，占据大部分视野；用户主要时间在看图传，其余信息为辅助。
- **信息层级**：图传上的叠加（准心、电容条、自瞄框等）为**一级信息**，与当前操作强相关；右侧面板为**二级信息**（状态、指令）；诊断、悬浮窗等为**三级**，需要时查看。
- **区域划分**：界面至少需明确以下**区域**（名称与职责），设计稿中请标出各区域的位置、占比、与相邻区域的关系：
  - **图传区**：居中大块，用于显示机器人实时图传；其上叠加 HUD（准心、条、框、文字等）。
  - **右侧主面板**：固定在一侧，承载“角色选择、诊断、赛事状态、等级与哨兵、态势/机器人列表、战术建议、赛事指令、插件”等；需明确**分组方式、标题与内容层级、是否可折叠/滚动**。
  - **状态栏**：窗口底部，显示连接与就绪状态等一行摘要信息。
  - **悬浮窗**：可独立于主布局的小块，用于基地底装甲、遥控器状态等；需明确**默认位置、是否可拖拽、是否可关闭/收起、与图传区或主面板的视觉从属关系**。

---

## 三、必须存在的界面元素清单（按区域）

以下为功能与内容清单，**不规定具体长什么样**，只规定“要有”；设计稿需为每一项安排**明确的位置、形态和层级**。

### 3.1 图传区（主区域）

- **底层**：机器人图传画面（无图传时占位表现，如“图传未连接”或统一占位图）。
- **叠加在图传上的元素**（需在设计稿中标出相对图传的位置，如左上/中上/中心/右下等）：
  - 自瞄状态（三态：离线 / 未瞄准 / 瞄准中）— 文字或图标。
  - 超级电容状态（类体力条）— 条状，需有位置与朝向。
  - 官方总能量/能量限制 — 若与电容条不同，需单独标出。
  - 己方/对方血量（或关键血量信息）— 在图传上的位置与形式。
  - 发弹量、热量限制 — 位置与形式（条或数字）。
  - buff 剩余时间与增益描述。
  - 准心（英雄/工程/云台为十字类，步兵为“扌”型）— 中心位置与样式。
  - 飞坡对准线（步兵，可开关）— 相对准心的位置（如准心两侧竖线）。
  - 自瞄识别框（紫框等）— 可多个，随目标移动。
  - 底盘姿态（如 unlocked/locked/low/high）— 在图传上的位置。
  - 受击提示、基地扣血/受击反馈 — 出现位置与样式（如闪红、图标）。
  - 比赛时间 — 建议图传上有一处明确时间显示（如右下）。
  - （步兵）跳跃目标、pitch/yaw 角度、自瞄目标距离、大小符开启提醒等 — 需标出与准心/发弹量/能量条的相对位置（见操作.md 步兵描述）。
- **中央强提示**：买活预警等需要用户立即注意的提示，应占据图传中央或显眼位置，与普通 HUD 区分开。

### 3.2 右侧主面板（需分组与层级）

- **角色选择**：英雄 / 工程 / 步兵1 / 步兵2 / 云台手；切换后下方内容或图传叠加可随角色变化，设计稿需说明“切换角色后哪些块显示/隐藏或变化”。
- **链路诊断**：MQTT 连接状态、MQTT 数据最近更新时间、图传数据最近更新时间；一键复制诊断信息。若还有“丢包/超时计数”，需占位。
- **赛事状态**：比赛阶段、剩余时间、血量（待协议接入）等；与“等级/性能体系”是同一组还是上下分组需明确。
- **等级与性能体系**：己方/对方等级、己方/对方性能体系（1/2/3）；主要给英雄视图用，其他角色是否显示需明确。
- **哨兵状态**：当前目标、选择状态、目的位置；主要给英雄视图用。
- **态势/小地图**：简化战场与机器人点位；与“机器人状态列表”是合并还是上下排列需明确。
- **机器人状态列表**：各机器人 ID、阵营、血量等。
- **战术建议**：血量对比、追击/撤退/谨慎建议、经济/买活提示等；视觉上需与普通状态区分（如强调色、区块）。
- **赛事指令**：性能体系 1/2/3、兑换发弹量、空中支援等按钮；需明确主次与误触防护（如二次确认）。
- **插件区**：为后续扩展留一块固定区域，可折叠或收缩。

设计稿需明确：各块**标题与内容的字体层级、间距、是否可折叠、在窄屏下的滚动或收缩方式**。

### 3.3 悬浮窗

- **基地底装甲是否打开**：单独小块，需有固定或可拖拽位置。
- **遥控器状态（丢控与否）**：可与基地底装甲同窗或分块，需标出。
- **全向感知提醒**（若做）：同上，需标出位置与触发状态。
- 设计稿需明确：悬浮窗**默认出现位置（如图传区四角或右侧边缘）、是否可拖拽、是否可关闭/最小化、与主窗口的层级关系**。

### 3.4 状态栏（窗口底部）

- 一行摘要：如“就绪 | DataCenter 已创建 | MQTT 已连接”等；设计稿需规定**内容范围与更新规则**，避免出现无关占位文案（如调试用文字）。

## 四、按角色的视图差异（设计稿必须区分）

- **英雄**：图传上叠加自瞄/电容/能量/血量/发弹/热量/buff、基地扣血与受击、前哨站与基地血量、准心、底盘姿态、受击、比赛时间等；右侧有**等级与性能体系、哨兵状态**；悬浮窗有基地底装甲、遥控器状态；飞坡线等可开关。
- **工程**：通用功能齐全，兵种特殊为机械臂姿态；布局可参考英雄，信息密度按工程需求调整（设计稿需给出与英雄的差异点）。
- **步兵1**：图传上**电容条（无目标时可长绿条）、自瞄状态（如 notarget）、底盘状态（unlocked/locked/low/high）、紫框自瞄框、扌型准心、飞坡线**；车辆状态类英雄；**跳跃目标**在底盘状态下方；**pitch 角度**在发弹量左侧/紫框右侧，**yaw 角度**在能量条右侧；**大小符提醒**在我方哨兵图标下；自瞄目标距离与高亮残血等合并展示。右侧面板是否与英雄共用或简化，需在设计稿中明确。
- **步兵2**：与步兵1 类似，可按队伍分工微调（设计稿标注与步兵1 的异同）。
- **云台手**：通用功能齐全，无额外兵种专属块；侧重态势与全局状态（设计稿需体现“信息多但不杂乱”的排布）。

设计稿应至少提供：**英雄、步兵1 两种视图的线框图或高保真**，其余角色可注明“同英雄”或“同步兵1 微调”。

## 五、交互与状态（设计稿需考虑）

- **角色切换**：切换后，右侧哪些区块显示/隐藏、图传上哪些叠加显示/隐藏，需在说明或标注中写清。
- **图传断流/无数据**：图传区占位表现（纯色+文案或占位图），叠加层是否保留、是否置灰或隐藏，需统一。
- **悬浮窗**：打开/关闭、拖拽范围、是否可贴边收起。
- **赛事指令**：点击后是否有 loading 或结果反馈；MQTT 未连接时是否禁用或弹窗提示，需在交互说明中注明。
- **诊断**：复制诊断信息后的反馈（如 toast 或状态栏短暂提示）。

---

## 六、需要设计稿明确的事项清单（交给设计师）

请在设计稿或说明中至少明确以下内容，便于开发按图实现：

1. **整体布局**：图传区与右侧面板的宽度比例、最小宽度；状态栏高度与内容规范。
2. **图传区**：无图传时的占位样式；各 HUD 元素（电容条、准心、自瞄框、飞坡线、时间、血量等）的**相对位置与大致尺寸**（可标注“左上”“中心偏下”等）。
3. **英雄视图**：右侧面板各区块的**顺序、分组、标题层级**；等级与哨兵、赛事状态、战术建议、赛事指令的视觉权重。
4. **步兵1 视图**：与英雄的差异；电容条、准心（扌型）、飞坡线、底盘状态、pitch/yaw、大小符、跳跃目标在图传上的**相对位置**。
5. **悬浮窗**：默认位置、尺寸、可拖拽范围、关闭/收起方式。
6. **颜色与状态**：连接正常/异常、买活预警、受击提示等使用何种颜色或样式区分（可沿用官方风格，但需写清）。
7. **响应式**：窗口缩小或不同分辨率下，右侧面板是否滚动、图传是否等比缩放、悬浮窗是否自动移位。
